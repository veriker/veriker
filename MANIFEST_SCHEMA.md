# manifest.json field reference

This file is the integrator-facing field reference for a Veriker audit bundle's `manifest.json`. One row per top-level key in the BundleManifest dataclass, with the columns external integrators (anyone reading the audit-bundle product cold) need to answer the three recurring questions:

1. What fields exist, and what is each one's type?
2. Which are required, and at which `schema_version`?
3. Where is each field validated — by which verifier step or plugin?

This doc is **not** a substitute for the canonical contract spec (the audit-bundle contract). Use this doc for the field-level shape; use the contract spec for the rules (`§C1..§Cn`) those fields enforce. If this doc disagrees with the `BundleManifest` class docstring at `audit_bundle/bundle_manifest.py` lines ~306-359, the docstring wins.

## How to read this table

Six columns per field:

- **top-level key** — the literal key in `manifest.json` (matches the BundleManifest attribute name).
- **type** — the Python annotation, rendered for an integrator (`dict[str, str]`, `tuple[dict, ...]`, `dict | None`, etc.).
- **required/optional at which schema_version** — one of `required (vcp-v1.1-canary4, legacy)`, `optional (W3+)`, `optional (Phase-0 substrate v0.2+)`, or `optional (v0.3 reservation — IGNORED)`.
- **validating plugin or verifier step** — the `_step_*` method in `audit_bundle/verifier.py`, the numbered `validate_manifest()` check in `audit_bundle/bundle_manifest.py`, or the TypedCheck plugin class that consumes the field. For v0.3 schema-reservation rows: the literal `v0.3 reserved — verifier IGNORES`.
- **sidecar-vs-inline** — every field is `inline (manifest.json top-level key)`. Three fields name sidecar paths (`decision_provenance_log`, `retrieval_trace_log`) or arrays of bundle-relative paths (`append_only_files`) — those rows say so.
- **one-line meaning** — lifted from the BundleManifest dataclass field comment / docstring entry.

v0.3 schema-reservation fields default to safe null / empty when omitted. The v0.3 substrate verifier does not read them; populating them is forward-compatible (v0.3 bundles round-trip cleanly through a v0.3 verifier with no behavior change, and the v0.4 verifier consumes the same shape this release reserves).

## Optional (v0.3 schema reservations; one field activated at v0.4)

Six fields landed at v0.3 wave 1 as schema reservations. At v0.4 the first reservation — `append_only_files` — is ACTIVATED (PRD `S_c9_1_prd.json`): the verifier reads, skips §C9 strict-SHA for declared paths, and dispatches `AppendOnlyAttributedCheck`. Two more have since gained core verdict-path consumers: `verifier_identity` (ARCH-02 structural step) and `causal_chain` (cross-host + assurance-profile guards; Layer-A crypto stays pilot-wired — see the rows). The remaining three (`rigor_profile`, `attested_serving`, `semantic_fidelity`) stay v0.3-IGNORE pending their respective v0.4 sprint work. Bundles that populate any reservation are forward-compatible (v0.4 plugin work consumes the same shape this release reserves). Defaults are safe null / empty so v0.2 / v0.2.1 / W3-baseline bundles continue to verify unchanged.

## Explicit callouts

These five facts are load-bearing for integrators and do not surface anywhere else as a single landing block:

1. **`payload/<name>.jsonl` files in a bundle are inspection artifacts and do NOT feed validation.** The verifier reads `manifest.json` top-level keys; it does NOT walk `payload/` looking for sidecar JSONL files to load. If you put `dispatch_records.jsonl` in `payload/` and DO NOT also populate the `dispatch_records` top-level array in `manifest.json`, the verifier will not see those records.
2. **`dispatch_records` is a TOP-LEVEL ARRAY in `manifest.json`** under the literal `"dispatch_records"` key. The presence of a sidecar `payload/dispatch_records.jsonl` is a human-inspection convenience only — it is not loaded by the verifier.
3. **Each `dispatch_records` entry carries its OWN `schema_version: "0.1"`** (the C15 record schema), distinct from the bundle's `schema_version: "vcp-v1.1-canary4"`. Records without `"0.1"` reject with `SCHEMA_VERSION_UNRECOGNIZED`. Do not conflate the bundle-level and record-level schema_version fields.
4. **`op_kinds_admitted` is a constructor argument on `DispatchRecordWellformedCheck`**, NOT a manifest field. Register the plugin with `DispatchRecordWellformedCheck(op_kinds_admitted=frozenset({...}))` to admit domain-specific op kinds beyond the default enum (RETRIEVAL / FORECAST / TOOL / COMPUTE / MODEL_CALL).
5. **All v0.3 schema-reservation fields default to safe null / empty** (`None` for `rigor_profile` / `attested_serving` / `verifier_identity` / `causal_chain` / `semantic_fidelity`; `()` for `append_only_files`) when omitted. Bundles emitting them traverse a v0.3 verifier with NO behavior change — the v0.3 verifier IGNORES the fields entirely. v0.4 ACTIVATION: `append_only_files` is consumed by the v0.4 verifier (`AppendOnlyAttributedCheck` plugin); bundles with empty `append_only_files == ()` continue to verify with NO behavior change (load-bearing back-compat invariant). The remaining five fields are still v0.3-IGNORE pending their v0.4 sprints. v0.4 plugin work consumes the same shape this release reserves; emitters that populate the fields at v0.3 are forward-compatible.

## Source of truth

- **Dataclass docstring** — `audit_bundle/bundle_manifest.py` lines ~306-359. If this doc disagrees with the docstring, the docstring wins for the W3-baseline + Phase-0 sections.
- **Contract semantics** — the audit-bundle contract §C1..§C20. Use the contract spec for rules; use this doc for the field-level shape.
- **Canonical emitter pattern** — `examples/<domain>_minimal/_build_bundle.py`. Every pilot in the `examples/` tree builds a manifest using the same `BundleManifest(...)` constructor; trace through one of the simpler pilots (`spectra_minimal`, `bom_minimal`) for the minimum required field set.
- **Verifier loader** — `audit_bundle/verifier.py::_load_manifest` (~lines 82-112). The function maps each `manifest.json` top-level key to its BundleManifest field; fields absent from the JSON take the dataclass default. v0.3 schema-reservation fields are NOT read by `_load_manifest` and round-trip via the dataclass default at v0.3.
- **v0.3 schema-reservation fields** — the six v0.3 schema-reservation fields (`append_only_files`, `rigor_profile`, `attested_serving`, `verifier_identity`, `causal_chain`, `semantic_fidelity`) each have a per-field reservation surface described in the rows above.
